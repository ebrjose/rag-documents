# Estado del arte para hacer RAGs en mayo de 2026

En mayo de 2026, el estado del arte para RAG ya no es solo:

Vector DB + embeddings + LLM

Eso ahora es el RAG básico. Lo moderno en producción es una arquitectura con búsqueda híbrida, reranking, metadata filtering, evaluación y, en casos avanzados, Agentic RAG o GraphRAG.

## Arquitectura recomendada

### 1. Ingesta inteligente

El flujo recomendado es:

Documentos → limpieza → extracción de tablas/OCR si aplica → chunks semánticos → metadatos → embeddings.

No conviene partir todo en chunks fijos de 500 tokens sin entender la estructura del documento.

Es mejor considerar:

- Secciones
- Títulos
- Páginas
- Tablas
- Fechas
- Permisos
- Fuente del documento
- Tipo documental
- Área o módulo relacionado

## 2. Búsqueda híbrida

El estándar práctico en producción es combinar:

BM25 / keyword search + vector search

Esto permite cubrir mejor los casos donde la búsqueda vectorial sola falla, por ejemplo:

- Nombres propios
- Códigos
- Fechas
- Números
- Siglas
- Términos exactos
- Nombres de expedientes
- Identificadores técnicos

Flujo típico:

query → BM25 + vector search → fusión de resultados → candidatos

Una técnica común para fusionar resultados es RRF, es decir, Reciprocal Rank Fusion.

## 3. Reranking

Después de recuperar documentos candidatos, se usa un reranker para ordenar mejor los chunks antes de enviarlos al LLM.

Flujo recomendado:

query → hybrid retrieval top 50/100 → reranker top 5/10 → LLM

El reranker es muy importante porque la primera búsqueda recupera candidatos, pero no siempre los ordena bien.

## 4. Query rewriting

Antes de buscar, el sistema puede reformular la pregunta del usuario.

Ejemplo:

Pregunta original:

"qué dice sobre mantenimiento?"

Consulta reformulada:

"recomendaciones de mantenimiento, mejora continua, operación, infraestructura y soporte técnico del sistema"

También se pueden generar varias consultas alternativas para mejorar la recuperación.

Esto ayuda cuando el usuario pregunta de forma ambigua, corta o incompleta.

## 5. Metadata filtering

Un RAG moderno no debería buscar en todo sin filtros.

Debe usar metadatos como:

- Fecha
- Autor
- Tipo de documento
- Área
- Proyecto
- País
- Módulo
- Permisos
- Nivel de confidencialidad
- Versión del documento
- Fuente original

Ejemplo:

Buscar solo en documentos del proyecto CRIFCAN, módulo AI, versión final, del año 2026.

## 6. Citations obligatorias

Una buena respuesta RAG debe mostrar de dónde salió la información.

No basta responder:

"El sistema recomienda aumentar la capacidad de GPU."

Debe responder algo como:

"El documento recomienda aumentar la capacidad de GPU para futuras fases, según la sección X del informe Y."

Idealmente debe incluir:

- Nombre del documento
- Página
- Sección
- Fragmento fuente
- Fecha o versión
- Link al documento si existe

## 7. Evaluación del RAG

No basta con probar manualmente.

Un RAG serio debe medir:

- Retrieval recall
- Precision@k
- Faithfulness
- Answer correctness
- Latency
- Cost per query
- Hallucination rate
- Número de documentos recuperados
- Calidad de las citas
- Feedback del usuario

También conviene guardar trazas:

- Pregunta original
- Pregunta reformulada
- Documentos recuperados
- Scores
- Resultado del reranker
- Contexto enviado al LLM
- Respuesta final
- Citas usadas
- Feedback del usuario

## Técnicas avanzadas

## 8. Agentic RAG

Agentic RAG significa que el sistema no hace un único retrieve fijo.

El agente decide:

- Si necesita buscar o no
- En qué fuente buscar
- Si debe reformular la pregunta
- Si debe hacer una segunda búsqueda
- Si debe consultar SQL
- Si debe consultar una API
- Si debe usar búsqueda vectorial
- Si debe usar búsqueda keyword
- Si debe verificar la respuesta
- Si debe pedir más contexto

Ejemplo de flujo:

Usuario pregunta → agente analiza → busca en documentos → detecta que falta contexto → reformula → busca otra vez → verifica → responde con citas.

Es útil para consultas complejas, pero puede ser más lento y caro.

No siempre es necesario para preguntas simples.

## 9. GraphRAG

GraphRAG usa grafos de conocimiento para representar relaciones entre entidades.

Sirve cuando los datos tienen relaciones fuertes, por ejemplo:

- Personas
- Áreas
- Direcciones
- Expedientes
- Normas
- Plagas
- Cultivos
- Países
- Proyectos
- Documentos
- Procesos
- Dependencias técnicas

Ejemplos donde GraphRAG ayuda:

- ¿Qué áreas están relacionadas con este proceso?
- ¿Qué documentos contradicen esta norma?
- ¿Qué proyectos dependen de este componente?
- ¿Qué plagas afectan a qué cultivos en qué países?
- ¿Qué funcionarios participaron en determinado flujo?
- ¿Qué módulos dependen del módulo de autenticación?

Pero GraphRAG no reemplaza al RAG normal.

Debe usarse cuando realmente hay relaciones importantes entre entidades.

## Stack práctico recomendado

Para un RAG serio on-premise:

### Almacenamiento

- PostgreSQL + pgvector
- Qdrant
- Milvus
- Elasticsearch
- OpenSearch
- MinIO para documentos originales

### Backend

- FastAPI
- Django
- Celery para ingesta
- Redis o RabbitMQ para colas
- PostgreSQL para metadatos
- OpenTelemetry para trazabilidad

### Retrieval

- BM25
- Vector search
- Hybrid search
- RRF
- Metadata filtering
- Reranking

### Modelos de embeddings

Opciones recomendables:

- bge-m3
- jina-embeddings-v3
- multilingual-e5
- e5-mistral
- Qwen embeddings

### Modelos reranker

Opciones recomendables:

- bge-reranker-v2-m3
- jina-reranker
- Qwen reranker

### LLM

Opciones recomendables:

- Qwen3
- Qwen2.5
- Llama 3.1
- Llama 3.3
- Gemma 3
- Modelos más pequeños si se busca baja latencia

## Flujo recomendado para producción

Un flujo sólido sería:

1. Usuario hace una pregunta
2. Se detecta intención
3. Se reformula la pregunta si es necesario
4. Se aplican filtros por metadatos
5. Se ejecuta búsqueda híbrida
6. Se fusionan resultados con RRF
7. Se aplica reranking
8. Se seleccionan los mejores chunks
9. Se arma el contexto
10. El LLM responde usando solo el contexto
11. Se generan citas
12. Se guarda la traza
13. Se registra feedback del usuario

Representación simple:

Usuario
→ Query rewriting
→ Metadata filtering
→ BM25 + vector search
→ RRF
→ Reranker
→ Context builder
→ LLM
→ Respuesta con citas
→ Evaluación y trazabilidad

## Recomendación directa

Para empezar bien en producción, implementaría esto:

1. Hybrid search: BM25 + embeddings
2. Reranker obligatorio
3. Metadata filtering por fecha, tipo, módulo, permisos y fuente
4. Citas obligatorias
5. Evaluación automática
6. Cache de embeddings y respuestas
7. Observabilidad de cada consulta
8. Agentic RAG solo para consultas complejas
9. GraphRAG solo si hay relaciones fuertes entre entidades

## Qué evitar

Evitar:

- Solo vector search
- Chunks fijos sin estructura
- No guardar metadatos
- No tener citas
- No evaluar respuestas
- Mandar demasiados chunks al LLM
- Usar Agentic RAG para todo
- Usar GraphRAG sin una necesidad real
- Confiar solo en prompts
- No controlar permisos por documento
- No versionar documentos
- No medir latencia ni precisión

## Conclusión

El RAG moderno no gana por usar más IA.

Gana por:

- Recuperar mejor
- Filtrar mejor
- Ordenar mejor
- Citar mejor
- Evaluar mejor
- Controlar permisos
- Mantener trazabilidad
- Reducir alucinaciones

La base recomendada para mayo de 2026 es:

Hybrid Search + Reranking + Metadata Filtering + Citations + Evaluation

Y sobre eso se puede agregar:

Agentic RAG para razonamiento complejo

GraphRAG para relaciones entre entidades
